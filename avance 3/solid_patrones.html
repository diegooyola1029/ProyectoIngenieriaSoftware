1. Principios de Diseño
PRINCIPIOS SOLID
Los principios SOLID son cinco guías de diseño orientado a objetos que mejoran la mantenibilidad, escalabilidad y legibilidad del código. A continuación se explica cada principio y cómo se aplica en el backend de Papitas Limoncitas.
S
Single Responsibility Principle
Principio de Responsabilidad Única
Definición: Una clase debe tener una única razón para cambiar — es decir, una sola responsabilidad.

Aplicación en Papitas Limoncitas: Cada clase tiene una responsabilidad clara y separada. PedidoController solo maneja las peticiones HTTP. PedidoService contiene la lógica de negocio. PedidoRepository maneja el acceso a la base de datos. Ninguna clase mezcla responsabilidades.
// ✅ Cada clase tiene UNA responsabilidad @RestController public class PedidoController { // ← solo HTTP private final PedidoService service; } @Service public class PedidoService { // ← solo lógica private final PedidoRepository repo; } @Repository public interface PedidoRepository // ← solo BD extends JpaRepository<Pedido, Long> {}
O
Open/Closed Principle
Principio Abierto/Cerrado
Definición: Las entidades de software deben estar abiertas para extensión pero cerradas para modificación.

Aplicación en Papitas Limoncitas: El sistema de precios usa una interfaz EstrategiaPrecio. Si se agrega un nuevo tipo de descuento (mayorista, temporada, etc.), se crea una nueva clase que implementa la interfaz, sin modificar el código existente de PedidoService.
// Interfaz base — cerrada para modificación public interface EstrategiaPrecio { double calcular(Pedido pedido); } // Extensión sin tocar lo existente public class PrecioMayorista implements EstrategiaPrecio { public double calcular(Pedido p) { return p.getTotal() * 0.85; } } public class PrecioNormal implements EstrategiaPrecio { public double calcular(Pedido p) { return p.getTotal(); } }
L
Liskov Substitution Principle
Principio de Sustitución de Liskov
Definición: Los objetos de una subclase deben poder sustituir a los de la clase base sin alterar el comportamiento del programa.

Aplicación en Papitas Limoncitas: UsuarioAdmin y UsuarioCliente extienden Usuario. Cualquier parte del sistema que use Usuario funciona correctamente con cualquiera de las subclases, sin comportamientos inesperados.
public class Usuario { public String getNombre() { return nombre; } public String getRol() { return rol; } } // Ambas subclases se comportan como Usuario public class UsuarioAdmin extends Usuario { ... } public class UsuarioCliente extends Usuario { ... } // Se puede usar Usuario sin saber cuál subclase es void mostrarInfo(Usuario u) { System.out.println(u.getNombre()); }
I
Interface Segregation Principle
Principio de Segregación de Interfaces
Definición: Los clientes no deben depender de interfaces que no usan. Es mejor tener muchas interfaces específicas que una general.

Aplicación en Papitas Limoncitas: En lugar de una interfaz gigante IServicio, se usan interfaces específicas por responsabilidad: IGestionPedidos, IGestionInventario, IGestionUsuarios. Cada servicio implementa solo lo que necesita.
// ❌ Mal — interfaz demasiado grande interface IServicioGeneral { crearPedido(); actualizarStock(); registrarUsuario(); generarReporte(); } // ✅ Bien — interfaces específicas interface IGestionPedidos { crearPedido(); cancelarPedido(); } interface IGestionInventario { actualizarStock(); verificarStock(); } interface IGestionUsuarios { registrar(); autenticar(); }
D
Dependency Inversion Principle
Principio de Inversión de Dependencias
Definición: Los módulos de alto nivel no deben depender de módulos de bajo nivel. Ambos deben depender de abstracciones.

Aplicación en Papitas Limoncitas: Spring Boot maneja la inyección de dependencias mediante @Autowired. PedidoController depende de la abstracción PedidoService, no de una implementación concreta. Esto facilita pruebas y cambios futuros.
@RestController public class PedidoController { // Depende de la ABSTRACCIÓN, no de la implementación @Autowired private PedidoService pedidoService; // Spring inyecta la implementación concreta automáticamente // Si cambia la impl, el Controller NO se modifica }
Resumen SOLID en Papitas Limoncitas: La arquitectura en capas (Controller → Service → Repository) garantiza naturalmente S, D e I. El patrón de estrategia de precios aplica O. La jerarquía de usuarios aplica L. Juntos, estos principios hacen que el proyecto sea mantenible y extensible.
2. Diseño de Software
PATRONES DE DISEÑO
Los patrones de diseño son soluciones reutilizables a problemas recurrentes en el desarrollo de software. Se clasifican en Creacionales, Estructurales y de Comportamiento. Papitas Limoncitas implementa varios de ellos de forma explícita o implícita.
Creacional Singleton
Garantiza que una clase tenga una única instancia y proporciona un punto de acceso global a ella. Útil para recursos compartidos como conexiones a base de datos o configuraciones.
Aplicación en Papitas Limoncitas Spring Boot gestiona los beans (@Service, @Repository) como Singletons por defecto. La conexión a MySQL a través de HikariPool es una única instancia compartida por toda la aplicación. También el EntityManagerFactory de JPA es un Singleton gestionado por el contenedor.
// Spring gestiona automáticamente el Singleton @Service // ← una sola instancia en toda la app public class PedidoService { ... } @Repository // ← una sola instancia compartida public interface ProductoRepository extends JpaRepository<Producto, Long> {}
Creacional Builder
Construye objetos complejos paso a paso, separando la construcción de la representación. Permite crear diferentes representaciones del mismo objeto.
Aplicación en Papitas Limoncitas La construcción de un Pedido con múltiples DetallePedido utiliza el patrón Builder implícitamente. En el frontend, el carrito de compras construye el objeto pedido de forma incremental antes de enviarlo al backend.
// Construcción de pedido paso a paso Pedido pedido = new Pedido(); pedido.setNombreCliente("Juan García"); pedido.setDireccion("Calle 10 #5-20"); pedido.setMetodoPago("Efectivo"); DetallePedido detalle = new DetallePedido(); detalle.setProducto(producto); detalle.setCantidad(3); pedido.getDetalles().add(detalle);
Estructural Repository (DAO)
Abstrae la capa de acceso a datos, separando la lógica de negocio de la persistencia. El repositorio actúa como una colección en memoria de objetos del dominio.
Aplicación en Papitas Limoncitas Cada entidad tiene su propio repositorio que extiende JpaRepository. PedidoRepository, ProductoRepository y UsuarioRepository abstraen completamente las operaciones SQL. El servicio no sabe si los datos vienen de MySQL, H2 o cualquier otra fuente.
@Repository public interface ProductoRepository extends JpaRepository<Producto, Long> { // Spring genera el SQL automáticamente List<Producto> findByStockGreaterThan(int minStock); Optional<Producto> findByNombre(String nombre); }
Estructural DTO (Data Transfer Object)
Objeto que transporta datos entre capas o sistemas, evitando exponer directamente las entidades del dominio y controlando qué información se comparte.
Aplicación en Papitas Limoncitas El README del proyecto menciona explícitamente "DTOs para entrada y salida de datos". Se usan para recibir datos del frontend y devolver respuestas sin exponer la estructura interna de las entidades JPA ni campos sensibles como la contraseña.
// Entidad — no se expone directamente @Entity public class Usuario { private String password; // campo sensible } // DTO — solo lo necesario para la respuesta public class UsuarioDTO { private String nombre; private String email; private String rol; // sin password ✅ }
Estructural MVC (Model-View-Controller)
Separa la aplicación en tres componentes: Modelo (datos), Vista (interfaz) y Controlador (lógica de flujo). Facilita el mantenimiento y la escalabilidad.
Aplicación en Papitas Limoncitas El proyecto sigue MVC claramente: el Modelo son las entidades JPA (Pedido, Producto, Usuario). La Vista es el frontend HTML/CSS/JS en static/. El Controlador son las clases @RestController que procesan las peticiones y devuelven respuestas JSON.
Comportamiento Strategy (Estrategia)
Define una familia de algoritmos, los encapsula y los hace intercambiables. Permite que el algoritmo varíe independientemente de los clientes que lo usan.
Aplicación en Papitas Limoncitas El sistema de cálculo de precios usa Strategy. El README menciona "Cálculo mayorista duplicado en backend". Se puede definir una estrategia de precio normal y otra mayorista, seleccionando cuál aplicar según el tipo de cliente o cantidad del pedido.
interface EstrategiaPrecio { double calcular(List<DetallePedido> items); } class PrecioNormal implements EstrategiaPrecio { public double calcular(...) { return subtotal; } } class PrecioMayorista implements EstrategiaPrecio { public double calcular(...) { return subtotal * 0.85; } }
Comportamiento Observer
Define una dependencia uno-a-muchos entre objetos: cuando un objeto cambia de estado, todos sus dependientes son notificados automáticamente.
Aplicación en Papitas Limoncitas Cuando el estado de un pedido cambia (pendiente → en ruta → entregado), el sistema puede notificar al administrador y actualizar el inventario. Spring Events o los eventos de JPA (@PostUpdate) implementan este patrón automáticamente.
@Entity public class Pedido { @PostUpdate // JPA Observer — se dispara al actualizar void onEstadoCambiado() { if ("ENTREGADO".equals(estado)) { // notificar admin, actualizar stock... } } }
Comportamiento Template Method
Define el esqueleto de un algoritmo en una clase base, dejando que las subclases redefinan ciertos pasos sin cambiar la estructura general.
Aplicación en Papitas Limoncitas El flujo de procesamiento de un pedido sigue siempre la misma secuencia: validar → calcular total → actualizar inventario → generar confirmación. JpaRepository de Spring Data aplica internamente Template Method para las operaciones CRUD estándar.

3. Diseño de Sistemas
ARQUITECTURA DE SOFTWARE
La arquitectura de software define la estructura fundamental del sistema: cómo se organizan sus componentes, cómo se comunican y qué decisiones técnicas los soportan. Papitas Limoncitas usa una arquitectura en capas con separación entre frontend y backend.
Estilo Arquitectónico
Arquitectura en Capas
El sistema se organiza en capas horizontales donde cada una tiene una responsabilidad específica. Las capas superiores consumen servicios de las inferiores pero no al revés. Es el estilo más usado en aplicaciones empresariales con Spring Boot.
Patrón de Comunicación
REST API
El frontend HTML/CSS/JS se comunica con el backend a través de endpoints REST. Las respuestas son JSON. Esto permite que el frontend y backend sean independientes y que en el futuro se pueda agregar una app móvil sin cambiar el backend.
Patrón de Datos
ORM con JPA/Hibernate
Las entidades Java se mapean automáticamente a tablas MySQL mediante JPA. Hibernate genera el SQL. Esto abstrae completamente la base de datos y permite cambiar de motor de BD con mínimo esfuerzo.
Despliegue
Monolito Modular
Todo el sistema se despliega como una sola unidad (JAR de Spring Boot). Es adecuado para el tamaño del proyecto. El README menciona Render como plataforma de despliegue y Railway para la base de datos en producción.

Flujo de una Petición REST

Ejemplo: Cliente hace un pedido desde index.html

1. El frontend JS recopila el carrito y hace POST /pedidos con JSON
2. PedidoController recibe la petición y valida el DTO
3. Llama a PedidoService.crearPedido()
4. El servicio valida stock con ProductoRepository
5. Crea el Pedido y sus DetallePedido en la BD mediante PedidoRepository.save()
6. Retorna el pedido creado como JSON al frontend
7. El frontend muestra la confirmación al cliente

Decisiones Arquitectónicas Clave

Decisión 1
Monolito vs Microservicios
Se eligió arquitectura monolítica modular sobre microservicios. Justificación: el equipo es pequeño (3 personas), el dominio es simple y el presupuesto es limitado. Un monolito es más fácil de desarrollar, depurar y desplegar para este contexto. Si el negocio crece, se puede migrar a microservicios por módulos.
Decisión 2
Frontend integrado vs Separado
El frontend HTML/CSS/JS se sirve desde Spring Boot como recursos estáticos. Esto simplifica el despliegue (un solo servidor) y evita problemas de CORS. La desventaja es que actualizar el frontend requiere reiniciar el backend, pero es aceptable para el tamaño del proyecto.
Decisión 3
ORM (JPA/Hibernate) vs SQL puro
Se usa JPA con Hibernate en lugar de SQL directo. Justificación: reduce código repetitivo, genera las tablas automáticamente (ddl-auto=update), y el equipo ya conoce Java. La desventaja es menor control sobre las consultas, pero aceptable para este volumen de datos.
Decisión 4
Seguridad: BCrypt para contraseñas
El README menciona migración automática a BCrypt al iniciar sesión. Las contraseñas heredadas en texto plano se hashean automáticamente. Esta decisión mejora la seguridad sin romper cuentas existentes, siguiendo buenas prácticas de almacenamiento de credenciales.
Decisión 5
Carrito en localStorage
El carrito de compras se persiste en localStorage del navegador en lugar de en el servidor. Justificación: evita crear sesiones en el backend, reduce la carga del servidor y el carrito sobrevive refrescos de página. Solo se persiste en la BD cuando se confirma el pedido.
Conclusión arquitectónica: Papitas Limoncitas implementa una arquitectura en capas clásica, apropiada para su tamaño y contexto. Las decisiones técnicas priorizan la simplicidad, la mantenibilidad y el costo reducido de infraestructura, alineadas con las restricciones reales del proyecto (equipo pequeño, cliente con presupuesto limitado, usuarios no técnicos).
